## 2. Backend architecture

| Item | Location / detail |
|------|-------------------|
| **Entry file** | `backend/src/server.js` вҖ” imports `app` and `config`, runs `app.listen(config.port)`. |
| **Express app** | `backend/src/app.js` вҖ” creates Express app, CORS, `express.json()`, static `uploads`, mounts routers, `errorHandler`; in production serves `frontend/dist` and SPA fallback. |
| **Routes** | `backend/src/routes/` вҖ” `auth.js`, `cv.js`, `public.js`, `ai.js`. No separate controllers; handlers are inline in route files. |
| **Controllers/Services** | No dedicated controllers or services layer. Business logic lives inside route handlers and in `backend/src/utils/` (e.g. `pdfTemplate.js`, `slug.js`). |
| **Prisma** | Client in `backend/src/lib/prisma.js` (singleton). Schema in `backend/prisma/schema.prisma`. Migrations: `backend/prisma/migrations/`. Commands: `npx prisma generate`, `npx prisma migrate deploy` (production). |

**Backend flow (high level):**

- `server.js` вҶ’ loads `app.js` вҶ’ `config.js` loads `backend/.env`.
- `app.js` mounts: `/api/health`, `/api/auth`, `/api/cv`, `/api/public`, `/api/ai`, and `/uploads` (static).
- Auth: JWT in `Authorization: Bearer <token>`; `requireAuth` in `middleware/auth.js` used on protected routes.
- If `NODE_ENV=production` and `frontend/dist` exists, Express serves it and SPA fallback for non-API routes.

## 3. Frontend architecture

| Item | Location / detail |
|------|-------------------|
| **Framework** | Vue 3 + Vite. |
| **Entry file** | `frontend/index.html` (references script) and `frontend/src/main.js` вҖ” creates Vue app, Pinia, router, i18n, mounts `#app`. |
| **Router** | `frontend/src/router/index.js` вҖ” `createRouter(createWebHistory())`, routes for Home, Login, Register, Dashboard, CVBuilder, PublicCV; `beforeEach` auth guard for `requiresAuth`. |
| **API service** | `frontend/src/services/api.js` вҖ” Axios instance with `baseURL: import.meta.env.VITE_API_URL \|\| "http://localhost:4000"` and request interceptor adding `Authorization: Bearer <token>`. Other services: `authService.js`, `cvService.js`, `aiService.js` (all use this `api`). |
| **Build output** | `frontend/dist/` вҖ” produced by `vite build` (default Vite output). |

**Frontend flow (high level):**

- `index.html` вҶ’ `main.js` вҶ’ `App.vue` with `<router-view>`.
- Router resolves route вҶ’ page component (e.g. CVBuilder, Dashboard).
- Pages use Pinia stores (`auth`, `cv`) and services (`api`, `authService`, `cvService`, `aiService`) to talk to the backend.

## 4. How frontend and backend communicate (API base URL)

- **Frontend** uses a single Axios instance from `frontend/src/services/api.js` with:
  - `baseURL: import.meta.env.VITE_API_URL || "http://localhost:4000"`.
- **Build time:** `VITE_API_URL` is baked into the frontend bundle at `vite build` (Vite only exposes `VITE_*` env vars).
- **Usage:** All API calls go to `baseURL + path` (e.g. `POST /api/auth/login`, `GET /api/cv`, etc.). Photo and upload URLs are built as `VITE_API_URL + path` (e.g. `/uploads/...`) in components; if `VITE_API_URL` is empty in production (same-origin), relative paths are used.
- **Backend CORS** allows only `FRONTEND_URL` (from `backend/.env`). For same-origin (Express serving frontend), set `FRONTEND_URL` to the public app URL (e.g. `https://cv.divsketch.com`).

## 5. How the frontend should be served in production (Nginx vs Express)

**Option A вҖ” Express only (single process):**

- Build frontend: `npm run build` (from root) or `npm run build` in `frontend/`.
- Run backend from **monorepo root**: `NODE_ENV=production npm run start`.
- Express serves `frontend/dist` as static files and returns `index.html` for non-API routes (see `app.js`). No Nginx required for serving the SPA; Nginx can still be used in front for HTTPS and reverse proxy.

**Option B вҖ” Nginx for static + proxy to backend:**

- Build frontend with `VITE_API_URL` set to your public API URL (or same origin).
- Nginx serves `frontend/dist` for `/` and proxies `/api` and `/uploads` to the Node backend (e.g. `http://127.0.0.1:4000`).
- Run only the backend (no static in Express): e.g. `NODE_ENV=production npm run start`; ensure `FRONTEND_URL` matches the site origin.

**Recommendation:** Either use Express-only (simplest on a small VPS) or Nginx in front for SSL and optional static serving; both are valid. For HTTPS, put Nginx (or another reverse proxy) in front of Node in both cases.

## 6. All environment variables required for production

**Backend (`backend/.env`):**

| Variable | Required | Description |
|----------|----------|-------------|
| `PORT` | No (default 4000) | HTTP port the backend listens on. |
| `NODE_ENV` | Yes (set to `production`) | Enables production behavior (e.g. serving `frontend/dist` when present). |
| `DATABASE_URL` | Yes | PostgreSQL connection string, e.g. `postgresql://user:password@host:5432/dbname?schema=public`. |
| `JWT_SECRET` | Yes | Secret used to sign/verify JWTs; use a long random string. |
| `OPENAI_API_KEY` | If using AI | OpenAI API key for translation/AI features. |
| `FRONTEND_URL` | Yes | Allowed CORS origin (e.g. `https://cv.divsketch.com`). Same as app URL when Express serves the frontend. |

**Frontend (build-time only; e.g. in CI or shell when running `npm run build`):**

| Variable | Required | Description |
|----------|----------|-------------|
| `VITE_API_URL` | No (default `http://localhost:4000`) | Backend base URL. Use `""` for same-origin when Express serves the frontend; otherwise set to full API URL (e.g. `https://cv.divsketch.com`). |

**Optional backend (e.g. for Puppeteer on some hosts):**

- `PUPPETEER_EXECUTABLE_PATH` вҖ” path to Chrome/Chromium if not auto-detected.

## 7. Correct command to start the backend server

- **From monorepo root (recommended for production if Express serves frontend):**
  ```bash
  NODE_ENV=production npm run start
  ```
  This runs `npm --workspace backend run start` вҶ’ `node src/server.js` from the `backend` workspace; NodeвҖҷs working directory is `backend/`, but `app.js` resolves `frontend/dist` as `backend/../frontend/dist`, so running from root with workspaces, the path is correct when you run `npm run start` from root (npm runs the script from `backend/`).

- **From backend directory:**
  ```bash
  cd backend
  NODE_ENV=production node src/server.js
  ```
  Or: `NODE_ENV=production npm run start` (from `backend/`).

- **Development:**
  ```bash
  npm run dev:backend
  ```
  (runs `nodemon` in `backend`).

---

## 8. Correct command to build the frontend

- **From monorepo root:**
  ```bash
  npm run build
  ```
  Runs `npm --workspace frontend run build` вҶ’ `vite build`. Output: `frontend/dist/`.

- **With production API URL (when not same-origin):**
  ```bash
  cd frontend && VITE_API_URL="https://cv.divsketch.com" npm run build
  ```
  Or from root (workspace script doesnвҖҷt pass env by default):
  ```bash
  VITE_API_URL="https://cv.divsketch.com" npm run build
  ```
  (Vite will see `VITE_API_URL` when run from root as well.)

- **Same-origin (Express serves frontend):**
  ```bash
  VITE_API_URL="" npm run build
  ```
  So the app uses relative URLs for API and uploads.

## 9. Recommended production deployment structure for a VPS

1. **Server:** One VPS; Node 18+ and PostgreSQL installed.
2. **App location:** e.g. `/home/divsketch-cv/htdocs` (CloudPanel site divsketch-cv; clone repo here).
3. **Process manager:** Use PM2 so the backend restarts on crash and on reboot:
   ```bash
   cd /home/divsketch-cv/htdocs
   npm install
   cd backend && npx prisma generate && npx prisma migrate deploy && cd ..
   # Build frontend (same-origin example)
   VITE_API_URL="" npm run build
   pm2 start ecosystem.config.cjs
   pm2 save && pm2 startup
   ```
4. **Reverse proxy:** Put Nginx (or Caddy) in front of Node:
   - Listen on 80/443; proxy all to `http://127.0.0.1:4000`.
   - Use Certbot for SSL (e.g. `certbot --nginx`).
5. **Env:** `backend/.env` with production `DATABASE_URL`, `JWT_SECRET`, `OPENAI_API_KEY`, `FRONTEND_URL` (e.g. `https://cv.divsketch.com`).
6. **Uploads:** Ensure `backend/uploads` exists and is writable by the user running Node.
7. **Single server:** Backend serves API + `frontend/dist` (one Node process, one port); Nginx only for SSL and proxy. No need to serve `frontend/dist` from Nginx unless you prefer it.

This gives a clear architecture overview, folder structure, backend/frontend flow, API base URL usage, production serving options, env vars, start/build commands, and a concise VPS deployment flow.
